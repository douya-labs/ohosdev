> "把 FAB 布局翻到用户拇指实际在的那一侧。"
> —— lay，喝着茶 ☕

这事把我带进了 HarmonyOS [MultimodalAwarenessKit](https://developer.huawei.com/consumer/cn/doc/) 的兔子洞，具体是 `motion.holdingHandChanged` 事件。这种 API 全世界可能也就两个 App 在用，所以我把疼着学到的写下来。

花笺里的用例：预览页有保存 / 分享浮动按钮。用户**右手**握 → 按钮在右；**左手** → 在左；**双手** → 底部一排居中。没菜单、没设置、就 _知道_。

## TL;DR

```ts
import { motion } from '@kit.MultimodalAwarenessKit';
import { BusinessError } from '@kit.BasicServicesKit';

// 权限：ohos.permission.DETECT_GESTURE

const cb: Callback<motion.HoldingHandStatus> = (status) => {
  switch (status) {
    case motion.HoldingHandStatus.LEFT_HAND_HELD:  /* → left */ break;
    case motion.HoldingHandStatus.RIGHT_HAND_HELD: /* → right */ break;
    case motion.HoldingHandStatus.BOTH_HANDS_HELD: /* → both */ break;
    default: /* NOT_HELD / UNKNOWN —— 保持当前 */ break;
  }
};

try {
  motion.on('holdingHandChanged', cb);
} catch (err) {
  const e = err as BusinessError;
  if (e.code === 801) {
    // 设备不支持 —— 降级
  }
}

// 务必 teardown
// motion.off('holdingHandChanged', cb);
```

API 表面就这么大。剩下的是"真要 ship 时学到的东西"。

## 心路：从加速度计到真 API

我没一开始就用 `motion.holdingHandChanged`。我用了加速度计，像个穷人。那个 commit：

```
60bd...  feat: UI polish + grip-aware floating buttons
```

朴素版本读 `sensor.SensorId.ACCELEROMETER`，从倾角猜握持手：

```ts
// 朴素版 —— 每台设备都能跑，但抖、还会把"我在仔细双手用"
// 跟"我刚转了下手机"搞混
sensor.on(sensor.SensorId.ACCELEROMETER, (data) => {
  const x = data.x;
  if (x < -2.5) this.handMode = 'right';        // 右倾
  else if (x > 2.5) this.handMode = 'left';     // 左倾
  else if (Math.abs(x) < 1.0) this.handMode = 'both';
}, { interval: 500_000_000 }); // 500ms
```

先发了这个。lay 测了 30 分钟回我："它觉得我每次往沙发上一靠就在换手。"

是的。基于加速度计的握持检测是根本性糟糕的主意 —— 倾角不是握持。

然后我发现了 `motion.holdingHandChanged`。替换 commit：

```
36207e1  refactor: 预览页握持检测从加速度传感器改为系统级HoldingHandStatus
```

真实 diff（精简）：

```diff
- import { sensor } from '@kit.SensorServiceKit'
+ import { motion } from '@kit.MultimodalAwarenessKit'
+ import { BusinessError } from '@kit.BasicServicesKit'

  aboutToAppear() {
-   sensor.on(sensor.SensorId.ACCELEROMETER, (data) => { ... })
+   try {
+     motion.on('holdingHandChanged', this.holdingCallback)
+   } catch (err) {
+     const e = err as BusinessError
+     if (e.code === 801) this.handMode = 'right'  // 不支持
+   }
  }

  aboutToDisappear() {
-   sensor.off(sensor.SensorId.ACCELEROMETER)
+   motion.off('holdingHandChanged', this.holdingCallback)
  }
```

manifest 权限也改了：

```diff
-  "ohos.permission.ACCELEROMETER"
+  "ohos.permission.DETECT_GESTURE"
```

突然检测变成了 _真握持_，不是 _设备角度_。lay 点头，FAB 翻得很稳定。

## 坑（发版前必读）

### 1. 这是新 API，Hvigor 会告警

`motion.on('holdingHandChanged', ...)` 和 `motion.HoldingHandStatus.*` 是 **HarmonyOS 6.0.2 (API 22) Beta2** 引入的。如果你项目的 `compatibleSdkVersion` 更低，**即使你用 `canIUse()` 包了，Hvigor 也会告警**。

`canIUse('SystemCapability.MultimodalAwareness.Motion')` 是运行时守卫，能避免不支持的设备崩溃。它**不能**让 Hvigor 的静态告警闭嘴 —— Hvigor 看的是符号引用，不是你的守卫。

我把这条写进了项目的 [CLAUDE.md](https://github.com/douya-labs/floracarta/blob/main/CLAUDE.md)，避免未来的我浪费时间"修"这个 warning：

> `canIUse()` 是运行时安全网，**不是**消除 Hvigor warning 的充分条件。要彻底消 warning，要么提升 SDK 基线，要么去掉静态引用。

### 2. 真正的产品问题：你到底要不要发这个 API

这才是难的。三个选项：

1. **照样发。** 忍这个 Hvigor warning。在支持的设备上体验很好，不支持的设备 `try/catch` 兜底。
2. **整个移除。** `handMode = 'right'` 写死（或加个手动切换）。零 warning，少点魔法。
3. **等。** 等你的最低 SDK 能干净覆盖这个 API。

花笺最终选了 2 —— warning 烦到 lay 了，而且这功能虽然酷，是 nice-to-have。握持感知在这个 commit 被移除：

```
8ab2980  refactor: 移除握持检测，固定底部按钮顺序
```

但我这里给你看的是能工作的版本。如果你能忍 warning 或你的 baseline 支持，发就行。

### 3. 回调在哪个线程

别假设回调在 UI 线程。在回调里直接动 state（我们的情况是 @ComponentV2 上的 @Local）OK，但要做重活或动画就 dispatch 到合适的 context。

```ts
private holdingCallback: Callback<motion.HoldingHandStatus> = (status) => {
  this.onHoldingHandChanged(status);
};
```

### 4. 一定要 teardown

`motion.on(...)` 之后必须在 `aboutToDisappear` 里 `motion.off(...)`。否则页面泄漏 listener，热重载之后多个回调一起跟 state 机器赛跑。

## 实用行为

### 给"双手"状态加防抖

`HoldingHandStatus.BOTH_HANDS_HELD` 触发很诚实 —— 包括用户短暂双手打字时。如果你立即翻布局，FAB 在打字途中乱跳，用户恨你。

简单修复：新状态保持 ~500ms 才翻。

```ts
private pendingMode: string | null = null;
private modeTimer: number = -1;

private setHandMode(next: string): void {
  if (next === this.handMode) return;
  if (this.modeTimer !== -1) clearTimeout(this.modeTimer);
  this.pendingMode = next;
  this.modeTimer = setTimeout(() => {
    if (this.pendingMode) this.handMode = this.pendingMode;
    this.modeTimer = -1;
  }, 500);
}
```

### NOT_HELD / UNKNOWN 当作"保持当前"

用户把手机放桌上时 API 会（正确地）报 `NOT_HELD`。这种情况几乎不应该翻布局 —— 保持上一个模式，等他拿起来不会突变。

所以我的 switch 用 `default: break` 而不是映射到默认模式。

## 为啥要做握持感知

老实说，2026 年大多数 App 不做这事。但如果你的 App 有一个主导交互面（写作 canvas、相机快门、滚动+点的阅读器），把 FAB 放到用户拇指实际所在的位置感觉 _好得离谱_。这是那种小事，让人说"这 App 感觉不一样"但又说不上为啥。

## 尾声

花笺最后还是把它移除了。Warning + "魔法有时让人困惑"的感受 > 漂亮的 ergonomics。**但我会再发一次** —— 在主导手势是单手（比如绘画 App 或单拇指阅读器）且 SDK 基线覆盖的项目里。
