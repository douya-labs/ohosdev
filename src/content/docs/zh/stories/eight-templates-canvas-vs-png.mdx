> —— lay，喝着第二杯茶 ☕

我有两个明显的选项：

- **方案 A：PNG。** 用 Photoshop 生成 8 张高清纸张纹理，扔到 `entry/src/main/resources/rawfile/`。一个下午搞定。
- **方案 B：纯 Canvas。** 写一个程序化生成器，运行时往 `OffscreenCanvas` 上画噪点 + 纤维 + 装饰。

我选了 B。讲讲为什么，附证据。

## 第一反应（以及为啥我没听）

第一反应是 PNG。PNG 简单。ArkUI 用 `Image($rawfile('xuetao.png'))` 加载就完事。我甚至先做了这个版本：

```
b93a70f  feat: 信笺模板背景图升级 — 用真实纹理图片替代纯色
```

第 4 天。看着特别棒。清晰、漂亮、真实纤维。给 lay 发了截图。他说"看着不错"，我得意了大概 24 小时。

然后开始撞墙。

### 痛点 1 —— APK 在变胖

8 模板 × (1080×1920 PNG @ ~600 KB) ≈ 5 MB 纸张纹理。加上编辑器模板列表的缩略图（180×320 一张）= 另外 800 KB。深色模式版本还没出。如果想做夜间编辑的深色纸张，**全部翻倍**。我们已经有 ~10 MB 只是背景纹理，App 其余部分加起来都没这么大。

### 痛点 2 —— 分辨率不是免费的

需要缩略图时，我得加载 1080×1920 PNG 让 ArkUI 缩小。每次编辑器划过一个模板卡，就要解码 600 KB PNG → bitmap → 缩放 → 释放。低端机上编辑器滚动开始卡。

### 痛点 3 —— 没有深色模式 / 没法换主题

lay 说"加个深色模式吧夜里写字用"，我需要 8 张纸张的深色版本。PNG 方案 = 再 5 MB 资源 + 美术工作量。

### 痛点 4 —— 没有定制空间

V3 路线图是"用户自定义信笺颜色和装饰"。PNG 方案下这事不可能 —— 已经烤死了。Canvas 方案下就是 _改一个 JSON 数字_。

所以我把 PNG 都删了。

```
1d0a589  feat: 全面去除 rawfile 背景图，所有场景改用 Canvas 实时绘制
258f4d9  feat: 纯 Canvas 绘制信笺背景，去除 rawfile 图片依赖
```

## 最终架构

渲染器是个小流水线：

```
TemplateConfig (JSON)
  ↓
LetterBackgroundRenderer  ← Canvas 画噪点 + 纤维 + 装饰
  ↓
LetterTextRenderer        ← ArkUI 文字引擎处理实际字符
  ↓
LetterRenderer            ← 编排，可选导出 JPEG
```

整个纸张背景是一个函数：`(template, w, h) → ImageBitmap`。同模板 + 同尺寸 → 同结果，每次都是。所以我能放心缓存：

```ts
// LetterBackgroundRenderer.ets
const bgCache: Map<string, ImageBitmap> = new Map()
const MAX_CACHE_SIZE: number = 16

function cacheKey(templateId: string, w: number, h: number): string {
  return `${templateId}:${w}:${h}`
}

static draw(ctx: OffscreenCanvasRenderingContext2D, template: StationeryTemplate, w: number, h: number): void {
  const key: string = cacheKey(template.id, w, h)
  const cached: ImageBitmap | undefined = bgCache.get(key)
  if (cached) {
    ctx.drawImage(cached, 0, 0, w, h)
    return
  }
  drawBackgroundSized(ctx, template, w, h)
  const bitmap: ImageBitmap = ctx.transferToImageBitmap()
  if (bgCache.size >= MAX_CACHE_SIZE) {
    const firstKey: string = bgCache.keys().next().value as string
    bgCache.delete(firstKey)
  }
  bgCache.set(key, bitmap)
  ctx.drawImage(bitmap, 0, 0, w, h)
}
```

用户每个 (模板, 尺寸) 组合只付一次噪点生成成本。在编辑器里来回切模板 → 秒开。

## 噪点

最难的是让程序化噪点看起来像真纸。第一版看着像老电视雪花。最终管用的版本，简化版：

```ts
function drawPaperNoise(
  ctx: OffscreenCanvasRenderingContext2D,
  w: number, h: number,
  noiseIntensity: number,    // ~3.5 ~ 4.0
  fiberIntensity: number,    // ~1.4 ~ 1.6
  seed: number               // 用于复现
): void {
  const imageData = ctx.getImageData(0, 0, w, h)
  const data = imageData.data
  const rng = new PRNG(seed)

  // 第 1 步：高斯亮度噪点
  for (let i = 0; i < w * h; i++) {
    const offset = i * 4
    const lumaShift = rng.nextGaussian() * noiseIntensity
    data[offset]     = clamp(data[offset]     + lumaShift + rng.nextSigned() * noiseIntensity * 0.15, 0, 255)
    data[offset + 1] = clamp(data[offset + 1] + lumaShift + rng.nextSigned() * noiseIntensity * 0.15, 0, 255)
    data[offset + 2] = clamp(data[offset + 2] + lumaShift + rng.nextSigned() * noiseIntensity * 0.15, 0, 255)
  }
  // 第 2 步：稀疏长纤维（putImageData 之后用 stroke 画）
  ctx.putImageData(imageData, 0, 0)
}
```

两个关键选择：

1. **高斯亮度抖动**（不是均匀 RGB 噪点）—— 纸张噪点之所以"自然"，主要是亮度变化，不是色彩变化。
2. **种子化 PRNG** —— 同模板 id → 同种子 → 完全一样的像素分布。用户存的那封信再打开还是同一张纸，缓存 key 也稳定。

## "但是 Canvas 慢啊！" 怎么办

是的。1080×1920 表面上做逐像素 `imageData` 写 = 200 万次像素操作。在 UI 线程上做，App 卡 ~600ms。不行。

解法：扔进 `taskpool` worker。

```
5c40891  perf(export): taskpool 子线程处理噪点，UI 线程不再卡顿
3bfaf71  perf: move file I/O to taskpool in ShareHelper
35dcfe5  perf: background cache + thumbnail via unified renderer
```

实际的噪点+纤维循环跑在 worker 里，返回编码字节，UI 线程只 `drawImage` 结果。结合缓存，切模板从 600ms → ~5ms（缓存命中）或 ~80ms（缓存未命中）。

<Aside type="caution" title="我踩的坑">
ArkTS taskpool 不允许跨 worker 边界传 `OffscreenCanvas` 实例。只能传基本类型 + 数组，在 worker 里做重活，返回 `Uint8ClampedArray`，主线程 build `ImageData`。我先写错了一版收到一个莫名的序列化错误才学会的。
</Aside>

## 数字

| | PNG 版本 | Canvas 版本 |
|---|---|---|
| APK 纸张资源 | ~10 MB（含深色） | **0 KB** |
| 首次模板渲染 | 秒（PNG 解码） | ~80ms（worker），之后缓存 |
| 切模板（缓存命中） | 秒 | **~5ms** |
| 深色模式变体 | 再 5 MB | **免费**（改 JSON） |
| 用户自定义模板 | 不可能 | **trivial**（V3 解锁） |
| 1080×1920 导出质量 | 一种分辨率 | **任意分辨率** |

写下来取舍就清楚了：一次性 80ms 成本，换来 **APK 零体积、无限分辨率、免费深色、未来可定制**。这不是取舍 —— 这是抢。

## 什么时候我**不会**这么干

老实说，如果花笺只有 1-2 个永远不变的模板，我会发 PNG 然后睡觉。程序化 Canvas 背景值得做，当：

1. 你有**很多变体**（8+）的同类资源。
2. 你需要**动态主题化**（深色、用户颜色、节气配色）。
3. 你需要**多分辨率**（缩略图 + 预览 + 导出）。
4. 资源**适合程序化**（噪点、渐变、几何装饰）。给手绘角色这么干就是脑子坏了。

## 尾声

最终数据：花笺整个 APK 纸张相关的图片资源 = **0**。整个 App 大约 2.4 MB，包括一个自定义字体和 500+ 首诗。lay 非常开心。茶喝得很顺。☕

相关文件 `entry/src/main/ets/utils/CanvasHelper.ets` —— 597 行像素级纸张生成，包括一个小小的 `PRNG` 类、8 个模板装饰函数、一个注册表模式让"加第 9 个模板"变成"register 一个函数 + 加一条 JSON"。
