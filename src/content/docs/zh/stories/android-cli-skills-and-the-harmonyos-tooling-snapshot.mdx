## TL;DR

- 🌱 2026-04-16 Google 官方发布 `android` CLI（preview），同时上线 Skills 仓库与 `docs` 知识库命令，整套面向 AI 编码 agent。
- 📋 鸿蒙这边今天也有官方 CLI —— `ohpm`、`hdc`、DevEco 的命令行构建 —— 但它们是**按职能切分**（包/设备/构建），目前没有打包成单一 "agent-first" 二进制，也没有官方 Skills 风格的模块化仓库或 `docs` CLI。
- 🎯 今天用 AI 写 ArkTS，套路还是老套路：上下文喂饱 + 自己维护 SKILL 文件。可参考 [让 AI 别再瞎编 ArkUI 装饰器](/zh/tips/stop-arkui-hallucinations/)。

## Google 这边发布了什么（事实）

**2026 年 4 月 16 日**，Android 开发者官方博客发文 "LLM flexibility, agent mode improvements, and more for Android Studio"。这次发布里，有三件事是新的，值得单独拎出来：

1. **统一的 `android` CLI**（preview 版本）。它把以前的 `sdkmanager`、`avdmanager` 以及一部分 `adb` 流程合并进一个二进制里，输出格式明确针对 AI agent 解析做了优化。Google 自己给的数据是：LLM token 用量降低约 **70%**，agent 完成任务速度提升约 **3 倍**（[Android Developers Blog](https://android-developers.googleblog.com/2026/01/llm-flexibility-agent-mode-improvements.html)，2026-04-16）。**这是 Google 自家的基准数字**，我引用，不为它背书。
2. **`android init` 命令安装 Android Skills** —— 一组模块化的"专家技能单元"，AI 编码 agent 可以按需加载。仓库公开在 [github.com/android/skills](https://github.com/android/skills)，格式开放到允许第三方贡献自己的 Skill。
3. **`android docs` 是一个知识库命令**，让 agent 能在写代码的时候实时查询 Android 官方文档，而不是只靠模型 2024 年的训练记忆。

CLI 开箱兼容 **Gemini、Claude Code、Codex**，目标平台覆盖完整 Android 表面：手机、平板、Wear OS、Android Auto、Android TV（[SiliconANGLE](https://siliconangle.com/2026/04/16/google-introduces-new-agentic-ai-ready-tools-resources-android-developers/)，2026-04-16）。[Thurrott](https://www.thurrott.com/dev/335014/google-announces-revitalized-android-cli-for-agentic-development) 把它形容为一次 "revitalized" CLI 重做 —— 官方定位很明确：**这是 Android Studio 的补充，不是替代**，给 agent 工作流用的（[官方文档](https://developer.android.com/tools/agents/android-cli)）。

事实层就这些。Preview，未公布 GA 时间。

## HarmonyOS 开发者今天有什么

换到这一边。这些工具我每周做 FloraCarta 都在用，下面是肌肉记忆 + 重读官方文档的结果。

- **DevEco Studio** 是主 IDE，基于 IntelliJ，GUI-first。6.0 版本（参见[华为官方 release notes](https://developer.huawei.com/consumer/en/doc/harmonyos-releases/deveco-studio-new-features-600)）加了不少 IDE 内置 AI 能力：ArkTS-aware 的代码补全、AI 对话面板、CodeGenie 系列辅助。要不要升，我之前写过：[DevEco 5.1 vs 6.0 怎么选](/zh/tips/deveco-5-1-vs-6-0-pick-guide/)。
- **`ohpm`** 是包管理 CLI。install / publish / search / uninstall 对接 OpenHarmony 的包仓库，干净直接。我天天在用。
- **`hdc`** （HarmonyOS Device Connector）是设备侧 CLI，定位类似 `adb`：列设备、推文件、装 HAP、shell 进设备、看日志。靠谱工具。
- **DevEco 的命令行构建工具** 也存在（`hvigor` 构建系统提供 CLI runner，可以从终端 headless 构建 HAP），CI 场景下挺顺手。

到 2026-05-14 为止，我没在官方渠道找到等价物的几样东西：

- **一个 agent-first 的统一 `harmonyos`/`deveco` 二进制**，把 SDK / 模拟器 / 设备工具一口气包起来，输出对 AI agent 友好。零件都在，但目前没有以"面向 AI agent"为定位打包发布的统一 CLI。
- **一个官方 "Skills" 仓库**，里头放模块化、官方手工 curate 的专家技能单元，让 AI 编码 agent 能 `init` 安装。社区有人在做 SKILL.md 风格的东西（我自己维护着 [harmony-app-dev](https://github.com/douya-labs/harmony-app-dev)），但 developer.huawei.com 上目前没有第一方版本，至少我没找到。
- **`android docs` 的等价物** —— 即一条 CLI 命令，能让 AI agent 在写代码时结构化地查询鸿蒙官方文档。developer.huawei.com 作为网站和搜索很完整，但**没有**一条文档化的 CLI 把它暴露成 agent 可调用的检索接口。

以上是我手上的信息。如果有华为 DevRel 看到这篇说"上个月我们发了你没看到"，请把链接发我，我立刻更新这篇文章。我宁愿被纠正，也不要写错。

## 一张对照表，只放事实

| 能力维度 | Android（截至 2026-04-16） | HarmonyOS（截至 2026-05-14） |
| --- | --- | --- |
| 主 IDE | Android Studio | DevEco Studio（最新 6.0） |
| IDE 内 AI 辅助 | 有（Studio 中的 Gemini / agent mode） | 有（DevEco 6.0 的 CodeGenie、AI 对话） |
| agent-first 统一 CLI 二进制 | `android`（preview） | _暂无官方发布_ |
| 包管理 CLI | `sdkmanager`（已并入 `android`） | `ohpm` |
| 设备 / 调试 CLI | `adb`（流程并入 `android`） | `hdc` |
| 模拟器 CLI | `avdmanager`（已并入 `android`） | DevEco 模拟器（GUI 为主，命令行可拉起） |
| 构建 CLI | `gradle`（+ `android` 的 agent 流程） | `hvigor` |
| 官方 Skills 仓库 | [android/skills](https://github.com/android/skills) | _暂无官方发布_ |
| agent 可调用的文档查询 | `android docs` | _仅有网页门户_ |
| 官方声明的 agent 兼容性 | Gemini、Claude Code、Codex | DevEco 6.0 内置助手；外部 CLI agent 需 BYO |

这张表是描述性的。我没有给列加权，也没有评判输赢。两个生态、两套产品策略、两组优先级。Android 这边出了一个 preview；HarmonyOS 这边出的是另一种形态的方案，重心在 IDE 上。

## 这对鸿蒙开发者今天意味着什么

讲实话，今天**你日常的活儿**几乎不变。还是打开 DevEco Studio，还是 `ohpm install`，还是半夜 `hdc shell` 追一个 Sendable 报错。Android 的发布动不到这些。

**会变的**是你工作流里"prompt 工程 + AI 结对编程"那一块，间接受影响 —— 因为你用的 AI（Claude Code、Cursor、Gemini、Codex 任选）针对 Android 会越来越聪明。Android 训练过的 Skills 现在已经存在了，鸿蒙这边目前还没有这种"开箱即用"的版本。

所以实操上，如果换我自己干，结论就是：**继续把"喂饱上下文"这件事做得更死。**

这周我自己更刻意做的几件事：

1. **手工维护一份 ArkTS / ArkUI 的 SKILL.md** 放在你 repo 根目录里。我接的每个鸿蒙项目根都放一份，里面列**真实存在的**装饰器集合、生命周期钩子、`@kit.*` 导入路径，再加一个"不要发明"清单。完整套路在 [让 AI 别再瞎编 ArkUI 装饰器](/zh/tips/stop-arkui-hallucinations/) —— 如果你在 vibe-code ArkTS，这是单条最重要的习惯。
2. **把官方文档链接钉进 prompt 里**。在没有 `harmonyos docs` 这种 CLI 之前，最便宜的替代方案就是把 3–5 条规范的 developer.huawei.com URL 塞进 system prompt，并要求 agent 引用或闭嘴。比你想象中管用。
3. **会话开始时把 `oh-package.json5` 和 `build-profile.json5` 抓到 agent 的 context 里**。Android 的 agent 可以问它的 CLI；我们的 agent 得我们手喂。结果一样，只是字打得多。
4. **明确告诉 agent 这边是 `hdc` 而不是 `adb`**。大多数 AI agent 在面对鸿蒙时会顺手猜 `adb` 语法，然后悄悄给出错的命令。要在 prompt 里说清楚是 `hdc`，并提示几个不同点（`hdc file send`、`hdc install`、`hdc hilog`）。

这些都不是新技巧。但 4 月 16 号那天，"AI 原生就懂 Android" 跟 "AI 需要被你教鸿蒙" 之间的缝稍微拉宽了一点点 —— 也就意味着"喂饱上下文"这种纪律，比上个月更值钱了一点点。
